# 语调识别方案调研

## 一、当前实现与局限

### 1.1 现有逻辑（`backend/services/distilhubert_service.py`）

- **实际未使用 DistilHuBERT**，仅用传统声学特征：
  - RMS 能量、过零率（ZCR）、短时能量方差、频谱重心（spectral centroid）
  - 通过固定权重线性组合得到 0~1 的「紧张度」得分
- **问题**：
  - 未用深度学习，对真实情绪/紧张/欺骗的区分能力有限
  - 未使用基频（F0），注释中的「基频」未实现
  - 权重为启发式，易受环境噪声、说话风格影响，泛化差
  - 与「语调」语义（情绪、压力、说谎相关紧张）关联弱

### 1.2 需求对齐

- 测谎场景需要：**紧张度/压力/情绪异常** → 映射为 `tone_score`（0~1），与说谎概率融合。
- 输入：16kHz 单声道音频片段（与现有 STT 一致）；输出：保持 `ToneResult(tone_score=...)` 接口。

## 二、可选方案对比

### 方案 A：emotion2vec+（推荐，中文/多语言场景）

| 项目 | 说明 |
|------|------|
| **来源** | HuggingFace: `emotion2vec/emotion2vec_plus_base`，ACL 2024 相关工作，中文社区常用 |
| **能力** | 9 类情绪：angry, disgusted, fearful, happy, neutral, other, sad, surprised, unknown |
| **输入** | 16kHz 语音，与当前 pipeline 一致 |
| **使用方式** | `pip install -U funasr modelscope`，FunASR 或 ModelScope pipeline |
| **模型规模** | base ~90M 参数；另有 seed（更小）、large（更强） |
| **与紧张度映射** | 将 angry / fearful / surprised 等视为高紧张，neutral / happy 视为低紧张，用概率加权得到 `tone_score` |
| **优点** | 效果优于多数开源 SER 模型；多语言/跨环境鲁棒；文档与示例齐全 |
| **缺点** | 依赖 FunASR/ModelScope，包体积和首次下载略大；需在 CPU 或与 STT 错峰用 GPU |

### 方案 B：HuBERT 中文情绪（xmj2002/hubert-base-ch-speech-emotion-recognition）

| 项目 | 说明 |
|------|------|
| **来源** | HuggingFace，基于 HuBERT，在 CASIA 中文情绪数据上微调 |
| **能力** | 6 类：anger, fear, happy, neutral, sad, surprise |
| **输入** | 16kHz 单声道 |
| **使用方式** | `transformers` + 自定义分类头或按标签映射紧张度 |
| **优点** | 与现有 `transformers` 技术栈一致；专注中文 |
| **缺点** | 依赖中文情绪数据集，泛化可能不如 emotion2vec+；需自行写推理与紧张度映射 |

### 方案 C：SpeechBrain wav2vec2 情绪（speechbrain/emotion-recognition-wav2vec2-IEMOCAP）

| 项目 | 说明 |
|------|------|
| **来源** | SpeechBrain，基于 wav2vec2，IEMOCAP 英文数据 |
| **能力** | 离散情绪分类，约 78.7% 准确率 |
| **输入** | 16kHz，库内可自动重采样 |
| **使用方式** | `pip install speechbrain`，`foreign_class` 或预训练 pipeline |
| **优点** | 集成简单，支持 GPU；文档完善 |
| **缺点** | 主要为英文数据，对中文语调泛化未知；需将情绪标签映射为紧张度 |

### 方案 D：Wav2Small（轻量级，CPU 友好）

| 项目 | 说明 |
|------|------|
| **来源** | HuggingFace: `dkounadis/wav2small`，Wav2Vec2 蒸馏到约 72K 参数 |
| **能力** | 维度情绪：arousal（唤醒）、dominance、valence（效价） |
| **输入** | 16kHz，5s 音频约 9ms CPU 推理 |
| **优点** | 极轻量，适合与 STT/LLM 同机部署、常驻 CPU；arousal 可直接对应紧张/激活度 |
| **缺点** | 非中文专门优化；需用 arousal（及可选 valence）映射为 `tone_score` |

### 方案 E：保留/改进传统特征 + 简单 F0

| 项目 | 说明 |
|------|------|
| **思路** | 在现有 RMS/ZCR/频谱重心基础上，增加基频（F0）估计（如 pyworld、crepe 或 librosa）及抖动（jitter）/ shimmer，再重新标定权重或用小型回归模型映射为紧张度 |
| **优点** | 无大模型依赖，延迟低，易控 |
| **缺点** | 上限仍低于端到端情绪模型；F0 在噪声或清音段不稳定，需鲁棒处理 |

## 三、推荐结论与实施建议

### 首选：**emotion2vec+（base 或 seed）**

- **原因**：当前项目以中文为主，且已有 16kHz 音频管线；emotion2vec+ 在中文和多语言 SER 上表现好，可直接用「高紧张情绪」概率合成 `tone_score`，与测谎需求对齐。
- **实施要点**：
  1. 使用 FunASR：`AutoModel(model="iic/emotion2vec_plus_base")`（或 `emotion2vec_plus_seed` 省资源），对现有 WAV 路径做 `generate(..., granularity="utterance", extract_embedding=False)`。
  2. 解析输出得到 9 类概率，定义紧张度权重，例如：  
     `tone_score = w_angry*P(angry) + w_fearful*P(fearful) + w_surprised*P(surprised) + ...`  
     其中高紧张类权重大，neutral/happy 权重小或为负。
  3. 将 `tone_score` 归一化到 [0,1]，封装为现有 `ToneResult(tone_score=...)`，无需改融合引擎。
  4. 若与 STT 共用 GPU，可只在非 STT 时段跑 emotion2vec（或使用 seed 在 CPU 上跑）。

### 备选：**Wav2Small（极度省资源时）**

- 需要最小依赖、CPU 常驻、与 STT/LLM 完全错峰时，可用 Wav2Small 的 arousal（和 valence）做线性或分段映射得到 `tone_score`，效果可能略逊于 emotion2vec+，但部署成本低。

### 不推荐

- **仅改进当前纯声学特征**：不引入情绪/压力模型的话，识别效果上限低，仍会「效果不太好」。
- **仅用英文 SER（如纯 IEMOCAP 模型）**：对中文语调泛化不足，除非后续加中文微调或仅作试验。

## 四、接口与集成方式（保持不变）

- **输入**：`analyze_tone(audio_path)` / `analyze_tone_from_b64(audio_b64)`，与现有 pipeline 一致。
- **输出**：`ToneResult(tone_score=float)`，融合引擎继续使用 `dimensions.tone`。
- 可在 `distilhubert_service.py` 中新增分支：若检测到 `funasr` / `emotion2vec` 可用则走 emotion2vec+，否则回退到当前声学特征实现，保证兼容。
